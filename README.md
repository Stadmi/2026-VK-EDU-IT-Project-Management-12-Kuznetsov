## 2. Приоритезация по MoSCoW

**Must have** (без этого никак):
*US1–US10* — все 10 историй. Это и есть костяк продукта: профиль студента, подбор вакансий, отклик, уведомления, регистрация работодателя, публикация вакансии, ранжирование кандидатов, защита данных. Без них платформа не работает.

**Should have** (сразу после запуска):
- Верификация навыков через тесты внутри платформы.
- Канбан-доска / воронка отбора для HR.
- Интеграция с LMS вуза (Moodle, балльно-рейтинговые системы).
- Дашборд для вуза по трудоустройству выпускников.

**Could have** (добавим при наличии времени и ресурсов):
- Рекомендации курсов для студента на основе нехватки навыков.
- Анонимный рейтинг студентов по направлениям.
- Платное продвижение вакансий в топ выдачи.
- Детальная аналитика по воронке найма для компаний.

**Won’t have (в MVP точно не берём)**:
- Отчёты под аккредитацию вуза.
- Сложная BI-система с прогнозами.

## 3. MVP и MLP

**MVP (что мы обязаны сделать в первой версии):**
- Студент: регистрируется, заполняет профиль (вуз, курс, навыки, проекты, оценки), получает список подходящих вакансий (мэтчинг по совпадению тегов навыков), откликается в один клик, получает базовые уведомления на почту.
- Работодатель: регистрирует компанию, создаёт вакансию с указанием требуемых навыков, видит отсортированный список кандидатов с процентом совпадения, может открыть профиль каждого.
- Система: простой алгоритм сопоставления навыков (ключевые слова + уровень), расчёт процента, настройки приватности профиля (скрыть оценки / проекты от работодателей).
- Всё это закрывает Must have (US1–US10).

**MLP (то, что добавим поверх MVP, чтобы пользователи полюбили продукт):**
- Онлайн-тесты для подтверждения навыков.
- Уведомления в мессенджеры (Telegram) в дополнение к почте.
- HR получает удобную воронку стадий отбора (отклик → тестовое → собеседование → оффер).
- Подключение вуза: автоматическая загрузка ведомостей из LMS, дашборд по трудоустройству.
- Подсветка дефицитных навыков у студента и предложение прокачивающих курсов.
- Анонимный рейтинг студентов по качеству профиля и востребованности навыков.

## 4. Детализация требований для двух главных историй из MVP

Я выбрал **US1** (профиль студента) и **US7** (ранжированный список кандидатов) – это самое ядро.

### US1 — Создание цифрового профиля студента

**Функциональные требования:**
- Форма регистрации: email, пароль, выбор роли «студент».
- Обязательные поля профиля: вуз (из выпадающего списка), факультет/специальность, курс, ожидаемый год выпуска.
- Навыки: нужно добавить минимум 3 штуки из предложенного облака тегов; если подходящего нет – дать возможность вписать свой. К каждому навыку выбрать уровень: Beginner, Intermediate, Advanced.
- Учебные проекты: можно прикрепить до 5 проектов, для каждого указать название, краткое описание, ссылку на GitHub/портфолио, технологии (например, Python, React).
- Оценки: либо вбить руками (предмет и оценка), либо загрузить файлик CSV/Excel.
- Верификация через корпоративную почту вуза: если почта заканчивается на домен вуза (универ.рф, .edu и т.п.), ставится метка «верифицированный студент».
- Кнопка «Сохранить» проверяет, что обязательные поля заполнены, подсвечивает пустые важные разделы.
- Настройки приватности: студент может скрыть от работодателей раздел с оценками и/или проектами.

**Нефункциональные требования:**
- При сохранении профиля интерфейс должен отреагировать не дольше чем за 2 секунды.
- Пароли хранятся в хешированном виде (bcrypt, 12+ раундов).
- Интерфейс доступен для людей с ограниченными возможностями (WCAG 2.1 AA).
- Загружаемые файлы (проекты, ведомости) не больше 10 МБ и проверяются на вирусы.
- Форма профиля должна быть доступна 99.9% времени в рабочие часы.

### US7 — Автоматический ранжированный подбор кандидатов

**Функциональные требования:**
- После создания вакансии работодатель видит список подходящих студентов, отсортированных по убыванию процента совпадения. Обновить список можно по кнопке «Обновить подборку».
- Процент совпадения считается по навыкам: сколько навыков из вакансии есть у студента. Порог – если совпадение меньше 50%, студент не показывается (но можно вручную снизить порог).
- Формула: точное совпадение ключевых навыков — 60% веса, смежные/похожие навыки — 20%, соответствие уровня — 20%. Администратор может менять веса.
- Карточка кандидата в списке: имя, курс, список навыков (те, что совпали, подсвечены), общий процент, кнопка «Открыть профиль».
- Фильтры: по вузу, курсу, наличию верификации.
- Если не нашлось ни одного кандидата с подходящим процентом, выдаётся сообщение: «Пока нет подходящих студентов. Попробуйте снизить порог совпадения».
- Экспорт текущей подборки в CSV: ID студента, имя, вуз, процент совпадения. Без личных данных и ссылок на профиль.

**Нефункциональные требования:**
- Подборка для 10 тысяч активных студентов должна формироваться не дольше 3 секунд.
- Если студент обновил навыки или работодатель изменил вакансию, результаты пересчитываются в течение 5 минут.
- Логи расчёта совпадения (какие навыки учтены, какой процент) хранятся 30 дней на случай споров.
- Файл экспорта должен быть подписан электронной подписью системы, чтобы его нельзя было незаметно подменить.
- Интерфейс подборки должен нормально выглядеть на мобильных устройствах и в современных браузерах (Chrome, Safari, Firefox).

# Практическое задание №3 — DDD, BDD, Wireframes, API

## 1. Доменные зоны (DDD)

Выделил 5 доменов, каждый отвечает за свою часть.

### 📁 Домен 1: Пользователи и роли (User Management)
**Что делает:**  
Регистрация, вход, управление ролями (студент, работодатель, представитель вуза), хранение базовых данных, настройки приватности.

**Глоссарий:**
- **Пользователь (User)** — любой зарегистрированный участник.
- **Роль (Role)** — студент / компания / вуз.
- **Верификация (Verification)** — подтверждение email через домен вуза или реквизиты компании.
- **Приватность (Privacy Settings)** — права, что показывать работодателю.

---

### 📁 Домен 2: Профиль студента (Student Profile)
**Что делает:**  
Хранение и редактирование детального портфолио студента: оценки, проекты, навыки, тесты, импорт из LMS.

**Глоссарий:**
- **Цифровой профиль (Digital Profile)** — расширенная информация студента.
- **Навык (Skill)** — технология или компетенция с уровнем (Beginner/Intermediate/Advanced).
- **Проект (Project)** — учебная/личная работа с названием, описанием, ссылкой.
- **Транскрипт (Transcript)** — список предметов и оценок.
- **Верификация навыка (Skill Verification)** — подтверждение через тест платформы.

---

### 📁 Домен 3: Вакансии и мэтчинг (Job & Matching)
**Что делает:**  
Публикация вакансий работодателем, автоматический подбор кандидатов по навыкам, отображение списков совпадений.

**Глоссарий:**
- **Вакансия (Vacancy)** — описание стажировки с требуемыми навыками.
- **Требование (Requirement)** — конкретный навык и желаемый уровень.
- **Мэтчинг (Matching)** — расчёт процента совпадения профиля студента с требованиями.
- **Релевантность (Relevance Score)** — число от 0 до 100%, показывает, насколько кандидат подходит.
- **Порог (Threshold)** — минимальный процент, при котором кандидат показывается (по умолчанию 50%).

---

### 📁 Домен 4: Отклики и коммуникации (Applications & Notifications)
**Что делает:**  
Отклик студента на вакансию, отслеживание статуса, уведомления обеим сторонам.

**Глоссарий:**
- **Отклик (Application)** — факт отправки профиля на вакансию.
- **Статус отклика (Application Status)** — новый / просмотрен / приглашение / отказ.
- **Уведомление (Notification)** — сообщение о новом совпадении, статусе и т.п.

---

### 📁 Домен 5: Аналитика и внешние интеграции (Analytics & Integrations)
**Что делает:**  
Дашборды для вузов, отчёты по трудоустройству, подключение LMS, экспорт данных.

**Глоссарий:**
- **LMS-интеграция (LMS Integration)** — автоматическая загрузка оценок и списков студентов из внешних систем.
- **Дашборд вуза (University Dashboard)** — графики и цифры по трудоустройству.
- **Отчёт (Report)** — выгрузка статистики в CSV/PDF.

## 2. BDD: Критический путь MVP

Выбрал путь студента: **поиск и отклик на подходящую стажировку**.  
Это ядро ценности — для студента важнее всего быстро найти релевантную позицию и не заполнять кучу бумаг.

### 🟢 Успешный сценарий

**Дано:**  
- Студент Иван зарегистрирован, его профиль заполнен (вуз, курс, навыки: Python — Intermediate, SQL — Beginner, Git — Intermediate).
- В системе есть вакансия «Python-стажёр» от компании с требованиями: Python (Intermediate+), Git (Beginner+).
- Порог мэтчинга установлен на 50%.

**Когда:**  
Иван открывает раздел «Подходящие стажировки».

**Тогда:**  
- Система показывает ему эту вакансию первой с процентом совпадения (допустим, 83%).
- Иван кликает «Откликнуться» — его цифровой профиль уходит работодателю.
- Иван видит уведомление «Отклик отправлен», а в его списке откликов появляется запись со статусом «новый».

---

### 🔴 Сценарий отказа (недостаточно навыков)

**Дано:**  
- Тот же Иван (Python, SQL, Git).
- Новая вакансия «Data Scientist intern» требует: Python (Advanced), Machine Learning (Intermediate).

**Когда:**  
Иван заходит в подборку стажировок.

**Тогда:**  
- Система не показывает эту вакансию, потому что совпадение ниже порога 50% (нет ML).
- Если Иван вручную снизит порог в фильтрах до 30% — вакансия появится, но с низким процентом.
- При попытке откликнуться система предупреждает: «Ваш профиль не соответствует ключевым требованиям, работодатель может отклонить отклик».  
  Отклик всё равно можно отправить, но шансов мало.

## 2. Уровень 2: Контейнеры (Container Diagram)

Раскладываем CareerConnect на основные программные контейнеры (сервисы, базы данных, приложения).
<img width="775" height="1367" alt="mermaid-diagram (1)" src="https://github.com/user-attachments/assets/ffb79135-4c5d-47af-a30c-3918d3d9ee98" />

**Контейнеры и их зоны ответственности:**
- **Web SPA (React)** — одностраничное приложение в браузере. Вся клиентская логика, экраны профиля, списка вакансий.
- **Backend API** — основной сервер. Обрабатывает запросы от клиента, авторизацию, бизнес-логику откликов, обращается к БД и очереди.
- **PostgreSQL** — реляционная база. Хранит пользователей, профили, вакансии, отклики.
- **Redis** — используется как очередь задач для пересчёта мэтчинга и как кэш горячих данных (например, подборки вакансий, чтобы быстро отдавать). В MVP можно обойтись без отдельного воркера, делая мэтчинг прямо в API, но очередь упрощает масштабирование.
- **Matching Worker** — отдельный сервис на Python (опционально в MVP). Слушает очередь из Redis, пересчитывает совпадения по вакансии при обновлении навыков студента. Позволяет вынести алгоритмы из основного API. В простейшем варианте можно оставить вычисления внутри API, но показал для полноты.

Взаимодействие:  
Студент жмёт «Подходящие стажировки» → Web SPA → запрос к Backend API → API читает из кэша Redis готовую подборку или быстро считает сам, обращаясь к PostgreSQL. При отклике создаётся запись в БД, и при необходимости отправляется email (через внешний сервис). Если студент обновил навыки, API кладёт задачу в очередь Redis, Matching Worker пересчитывает его совпадения и сохраняет результат в БД (или кэш).

## 3. Технологический стек MVP

Выбирал, исходя из того, чтобы быстро собрать работающий прототип, без лишней сложности, и чтобы команде (даже небольшой) было комфортно.

| Компонент       | Технология          | Почему выбрано |
|-----------------|---------------------|----------------|
| **Фронтенд**    | React (CRA или Vite) | Большое сообщество, множество готовых компонентов. Легко верстать интерфейс, SPA идеально подходит под дашборды и формы. Можно быстро переиспользовать код, если позже захотим React Native для мобилки. |
| **Бэкенд API**  | Node.js + Express (либо Python + FastAPI) | Решили остановиться на Node.js, потому что фронт на JS — единый язык, меньше контекстного переключения. Express простой, проверенный. Но если бы AI-мэтчинг требовал тесной интеграции с ML-библиотеками, выбрали бы Python/FastAPI. В MVP допустимо быстрое прототипирование на Node. |
| **База данных** | PostgreSQL           | Лучшая опенсорсная реляционка. Чёткая схема, транзакции, поддержка JSON для гибких профилей. Справится с нагрузкой даже при росте. |
| **Кэш/очередь** | Redis                | Лёгкий, как швейцарский нож: и кэш, и очередь. Для MVP можно запустить в одном докер-контейнере. Без него пока можно обойтись (хранить всё в памяти), но лучше сразу заложить, чтобы не переписывать логику откликов. |
| **Мэтчинг**     | Python (в отдельном воркере) | Так как алгоритм может усложняться (NLP, взвешивание), Python — стандарт для ML. В первой версии можно сделать обычным скриптом на Node.js, обрабатывающим совпадение строк, но задел на Python даёт гибкость. |
| **Инфраструктура** | Docker + Docker Compose | Все сервисы (API, БД, Redis, воркер) запускаются одной командой. Удобно для разработки и деплоя на любую VPS. |
| **Деплой**      | VPS (например, Yandex Cloud, Timeweb) | На старте не нужен Kubernetes. Простой сервер с Docker Compose дешевле и проще. Потом можно переехать в облачную managed-инфраструктуру. |

---

В итоге получаем типовой современный стек, который быстро собирается, знаком большинству разработчиков и не перегружает MVP.

## 2. Development Framework — Scrum

Выбрали **Scrum** спринтами по 2 недели.  
Три тезиса, почему он подходит нашему продукту:

1. **Быстрая обратная связь от рынка.** MVP должен проверять гипотезу «AI-мэтчинг соединит студентов и работодателей». Жёсткие итерации дают возможность каждые две недели показывать новую версию пилотным пользователям, собирать фидбек и сразу же корректировать бэклог. Если бы мы работали по Waterfall, то до первого юзера могли бы пилить полгода — слишком рискованно.

2. **Прозрачность для инвесторов и вузов.** Спринты, обзоры и ретро — это готовые артефакты. Можно честно сказать: «Через 2 недели покажем поиск с фильтрацией, через 4 — первый AI-подбор». Пилотные вузы видят прогресс, а не просто обещания.

3. **Снижение риска переусложнения.** Мы стартуем с минимальным AI (простое сравнение тегов). Scrum заставляет команду фокусироваться на Sprint Goal и не уходить в золочение — «а давайте сразу нейросеть обучим». Сначала — работающий скелет, потом — улучшения.

## 3. Team Rituals — что и когда встречаемся

Все ритуалы завязаны на спринт 2 недели. Команда распределённая, поэтому всё в Zoom/Telegram.

| Ритуал | Цель | Частота / Длительность |
|--------|------|------------------------|
| **Daily Standup** | Синхронизироваться, не блокировать друг друга. Каждый отвечает на три вопроса: что сделал вчера, что буду делать сегодня, есть ли блокеры. Без углубления в детали — для этого отдельные созвоны. | Ежедневно, кроме понедельника (тогда планирование). 15 минут максимум. |
| **Sprint Planning** | Распланировать бэклог на спринт. PO приносит приоритезированные истории, команда оценивает их в story points (покер планирования), договаривается, какой объём берём. | В первый день спринта (понедельник). 1 час. |
| **Sprint Review** | Показать заинтересованным сторонам (пилотный вуз, условный HR) работающий инкремент. Собрать обратную связь: «вот это ок, а тут неудобно». PO записывает идеи в бэклог. | Последний день спринта (пятница). 30–45 минут демо + обсуждение. |
| **Sprint Retrospective** | Посмотреть, как работалось внутри команды: что получилось хорошо, что можно улучшить в процессах. Выбрать 1–2 конкретных улучшения на следующий спринт. | Сразу после Review, пока эмоции свежи. 30 минут. |
| **Backlog Refinement** | PO и разработчики уточняют детали предстоящих историй: прорабатывают acceptance criteria, обсуждают технические риски, чтобы на Planning не спотыкаться. | Раз в неделю (среда), 45 минут. |

Такой ритм держит команду в тонусе и не даёт утонуть во встречах — суммарно ритуалы занимают не больше 10% рабочего времени.
